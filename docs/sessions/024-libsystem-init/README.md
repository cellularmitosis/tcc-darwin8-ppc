# 024 — toward libSystem-state init in tcc-linked execs

Session [023](../023-macho-executable/README.md) got `tcc -o exe`
producing standalone Mach-O executables that work for syscall-only
programs but crash (`SIGBUS`) when they touch libSystem-internal
state (printf, puts, malloc, getenv).

This session pushes further on three fronts:

1. **External-data PIC indirection now works in EXE mode** — was OBJ-only.
2. **Auto-injected `__tcc_start_main`** — tcc emits a tiny C function
   into every executable that initializes libSystem globals before
   calling main.
3. **`__mh_execute_header` synthesized in symtab** — the magic symbol
   dyld uses to identify the executable image when libSystem looks up
   `_environ` / `_NXArgc` / etc.

Result: many more libSystem patterns work. But the core mystery
remains: **dyld is not calling libSystem's `__mod_init_func`
initializers when our executable loads**, so libSystem's stdio
internals (`__sF`, `_environ_pointer.2733`) stay uninitialized.
gcc-built binaries get those initializers; ours don't.

## What works now via tcc-link

| Program | Result |
|---|---|
| `int main(){return N;}` | ✓ |
| `int sq(int n){return n*n;} int main(){return sq(7);}` | ✓ exit 49 |
| `write(1, "hi\n", 3)` (direct syscall) | ✓ prints, exit 0 |
| `getpid()` | ✓ |
| `strlen` / `strchr` (pure-fn libSystem) | ✓ |

## What still doesn't work

| Program | Symptom |
|---|---|
| `puts("hi")` | SIGBUS after `fstat(1)` (deref of uninitialized `__sF`) |
| `printf("hi %d", 42)` | same |
| `getenv("HOME")` | returns NULL (libSystem looks at its own `_environ_pointer`, not ours) |
| `malloc(N)` | likely SIGBUS (heap not set up) |

Same programs work fine when linked via `gcc-4.0` (which bundles
`/usr/lib/crt1.o` and triggers libSystem init).

## What was added

### `tcc/ppc-gen.c` (+10 lines)

PIC indirection for extern data symbols now fires for `TCC_OUTPUT_EXE`
in addition to `TCC_OUTPUT_OBJ`. The codegen emits the same
`addis r30, ha(slot - anchor) ; lwz lo(slot - anchor)(r30)` pattern,
and the EXE writer resolves the SECTDIFF-style offsets against
absolute slot vmaddrs.

### `tcc/ppc-macho.c` (+427 lines)

- **`exe_resolve_section_relocs`** now handles `R_PPC_HA16_PIC` /
  `R_PPC_LO16_PIC` by looking up the per-function PIC anchor and
  computing `(slot_vmaddr - anchor_vmaddr)` split into HA16/LO16
  halves.
- **`__nl_symbol_ptr` slot allocation extended** for extern data:
  function-stub slots come first, data slots follow.
- **Indirect symbol table extended** with one entry per data slot.
- **`__data` segment section now emitted** when the user code has
  initialized globals.
- **All globally-visible defined symbols** (not just `_main` and
  `_start`) now appear in the executable's symtab so dyld can find
  them. This is what gets `_NXArgc` / `_NXArgv` / `_environ` exported.
- **`__mh_execute_header`** added as an N_ABS external pointing at
  the `__TEXT` segment base.
- **`LC_TWOLEVEL_HINTS`** load command emitted (with zero hints) —
  matches the gcc-4.0 output exactly.

### `tcc/tccelf.c` (+39 lines)

`tcc_output_file` for PPC `TCC_OUTPUT_EXE` auto-injects this C source
via `tcc_compile_string` before dispatching to `macho_output_exe`:

```c
int    NXArgc  = 0;
char **NXArgv  = 0;
char **environ = 0;
int  main(int argc, char **argv, char **envp);
void exit(int);
void __tcc_start_main(int argc, char **argv, char **envp){
    NXArgc  = argc;
    NXArgv  = argv;
    environ = envp;
    exit(main(argc, argv, envp));
}
```

The crt-shim entry point (in `__text`, generated by the EXE writer)
sets up the stack and calls `__tcc_start_main` instead of `main`
directly. `__tcc_start_main` is a regular tcc-emitted function; its
references to `NXArgc` etc. go through normal codegen, and its calls
to `exit` go through normal extern-stub dispatch.

### `tcc/lib/lib-ppc.c` (+36 lines)

Documentation comment for the auto-injected `__tcc_start_main`
(the actual definition lives inline in `tcc_output_file`; the
comment here explains the relationship).

## The remaining mystery

Comparing our binary to a gcc-4.0 link of the same source:

```
$ DYLD_PRINT_INITIALIZERS=1 /tmp/h-tcc       (our linker)
... no init function calls printed ...
SIGBUS

$ DYLD_PRINT_INITIALIZERS=1 /tmp/h-gcc       (gcc-4.0 link)
dyld: calling initializer function 0x9010ba64 in /usr/lib/libSystem.B.dylib
dyld: calling initializer function 0x9017d330 in /usr/lib/libSystem.B.dylib
dyld: calling initializer function 0x9017e54c in /usr/lib/libSystem.B.dylib
hello via tcc-link!
exit=0
```

Both binaries have `LC_LOAD_DYLINKER /usr/lib/dyld`, `LC_LOAD_DYLIB
/usr/lib/libSystem.B.dylib`, the same MH flags
(`MH_NOUNDEFS|MH_DYLDLINK|MH_TWOLEVEL`), `LC_TWOLEVEL_HINTS`,
`LC_DYSYMTAB`, the right indirect-symtab entries.

What gcc has that we don't:

* `__DATA,__dyld` section (8 bytes, two function pointers dyld writes).
* `__DATA,__bss` and `__DATA,__common` segments.
* `MH_SUBSECTIONS_VIA_SYMBOLS` flag (a linker hint, shouldn't affect
  runtime).
* The crt1 code (which calls a chain of `_dyld_func_lookup`-resolved
  helpers — `__dyld_make_delayed_module_initializer_calls`,
  `__init_keymgr`, etc.).

The most likely candidates for the "missing trigger" are the
`__dyld` section and/or the explicit calls to dyld helpers from
within crt1. Worth experimenting with in a follow-up.
